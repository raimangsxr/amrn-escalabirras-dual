# Context Pack: 007-angular-20-migration

## Goal

Bump the Angular frontend from v16 to v20 while preserving the
existing NgModule + RxJS + zone.js + Karma/Jasmine architecture.
This is a tooling / dependency / builder migration, not an
architectural rewrite.

## Relevant Contracts

- `specs/contracts/frontend-angular/contract.md` (updated —
  Angular 16 → Angular 20; builder names reflect v20).

## Current Understanding

- The frontend is an Angular 16.1 SPA under `src/`. Bootstrap is
  the classic `platformBrowserDynamic().bootstrapModule(AppModule)`.
- All components are declared in `src/app/*.module.ts` files
  (`AppModule`, `AuthModule`, etc.). No standalone components.
- State is held in `BehaviorSubject`s inside `AuthService` and
  `AppService`. Components consume the streams via the `async`
  pipe.
- `tsconfig.json` declares `target: "ES2022"`, `module: "ES2022"`,
  `experimentalDecorators: true`, `strict: true`. These remain
  valid for v20.
- `angular.json` uses the legacy `@angular-devkit/build-angular:
  browser` builder. v20 removes that builder in favor of
  `@angular/build:application` (or its devkit equivalent).
- `@angular/material` is on v16. The legacy
  `@angular/material/prebuilt-themes/indigo-pink.css` prebuilt
  stylesheet is imported in `angular.json` and rendered in the
  bundle. Material 20 ships the same path but the prebuilt is
  deprecated; the documented replacement uses the M3 `mat.theme`
  mixin. Both are acceptable for this change; we keep the prebuilt
  for the smallest possible diff and document the deprecation in
  the contract.
- `package.json` has zero `devDependencies` related to Karma +
  Jasmine setup beyond `@types/jasmine`, `jasmine-core`, `karma`,
  `karma-chrome-launcher`, `karma-coverage`, `karma-jasmine`,
  `karma-jasmine-html-reporter`. v20 supports Karma natively
  through `@angular/build:karma`; we add it to
  `devDependencies` and remove the legacy
  `@angular-devkit/build-angular` Karma builder from
  `angular.json`.
- The frontend has zero unit tests today. The `test` target exists
  for parity with the rest of the Angular ecosystem but is not run
  in CI. We keep the target.
- The production `environment.prod.ts` ships with `apiBaseUrl:
  ''` (per `006-production-readiness`). This stays.
- The responsive iframe behavior introduced in `005-responsive-
  iframe` depends only on `ResizeObserver` + Tailwind arbitrary
  `clamp()`. Both are independent of the Angular version. No
  regression expected.
- The v18 migration added `provideHttpClient(withInterceptorsFromDi())`
  alongside the legacy `HTTP_INTERCEPTORS` provider in
  `AppModule`. Both coexist; the legacy interceptor still fires.
- The v19 migration added `standalone: false` to all components.
  No behavior change (was implicit before; now explicit).
- The v20 migration set `moduleResolution: "bundler"` in
  `tsconfig.json`. No code changes needed.
- `js-confetti ^0.11.0` is imported as a default export. v20 + TS
  5.8 + `esModuleInterop: true` continues to compile it.
- `tailwindcss ^3.3.3` is unchanged. v20 does not require Tailwind
  v4.

## Constraints

- **NgModules stay.** No standalone components, no
  `bootstrapApplication`. `AppModule` remains the bootstrap
  module.
- **RxJS + `BehaviorSubject` stay.** No signals.
- **`zone.js` stays.** No `provideZonelessChangeDetection`.
- **Karma + Jasmine stay.** No Jest, no Vitest, no Web Test
  Runner.
- **No SSR, no prerendering, no hydration.**
- **No visual changes.** Templates, classes, copy stay.
- **Material 20 prebuilt is acceptable** for v20 even though it is
  deprecated; switching to the M3 `mat.theme` mixin is out of scope.
- **Pin with `~`.** Avoid floating to a new minor accidentally.
- **Node ≥ 20.11** is required by Angular 20. The current dev env
  is on a current macOS with a current Node; no `.nvmrc` change
  expected.

## Risks

- **Peer dep conflicts at `npm install`.** Angular 20 may pull in a
  transitive dependency that conflicts with the existing Tailwind /
  PostCSS versions. Mitigation: pin Tailwind / PostCSS if
  necessary, or use `--legacy-peer-deps` only as a last resort.
- **Manual application-builder migration.** The
  `use-application-builder` schematic bundled with v20 requires
  Node ≥ 22.22.3 (the current dev env runs 22.14.0). The schematic
  is a static rewrite only, so we performed the migration by hand:
  - `build.builder` switched from
    `@angular-devkit/build-angular:browser` to
    `@angular/build:application`.
  - The `main` option renamed to `browser` (application builder
    schema requirement).
  - `buildOptimizer` and `vendorChunk` removed from
    `development` config (no longer in the application builder
    schema).
  - `assets` rewritten as `{ glob, input, output }` objects (the
    new schema rejects the legacy string-list format).
  - The legacy `@angular-devkit/build-angular` was dropped from
    `devDependencies`; `@angular/build` is declared directly.
  The migration result is identical to what the schematic would
  have produced.
- **TypeScript 5.8 strictness.** Angular 20 supports `~5.8.0`.
  Existing code is strict-mode-clean per the current `tsconfig`;
  the bump should not add new errors. If it does, treat each as a
  blocker and resolve before merging.
- **Builder config rename (`browserTarget` → `buildTarget`).**
  v17 deprecated `browserTarget`; v20 removes the legacy name.
  We update `angular.json` mechanically.
- **Material 20 theming deprecations.** The prebuilt theme is
  still present in Material 20 but logs a deprecation warning.
  Acceptable for this change; a future change can migrate to the
  M3 `mat.theme` mixin.
- **`@angular/build` vs `@angular-devkit/build-angular`.**
  Angular 20 ships both. We pick the one that the Angular CLI's
  `ng update` recommends (`@angular/build:application` is the
  v17+ canonical name). If the workspace requires the devkit
  variant, we use that.
- **No frontend tests to validate.** We rely on the typecheck +
  build + manual smoke. The contract's "No frontend unit tests"
  invariant is preserved.
- **No CSS regressions expected.** v20 ships zero changes to the
  CSS Custom Properties cascade or to Tailwind's contract.
- **`tslib` bump.** `tslib` will move to a newer minor. We accept
  the bump.

## Recommendations (do not create in this change)

- **Migrate to Material 20 M3 `mat.theme` mixin.** The prebuilt
  is deprecated; a future change can move to the M3 design token
  system.
- **Adopt standalone components.** A future change can migrate
  `AppModule` → `bootstrapApplication`, `provideRouter`,
  `provideHttpClient`, etc., component by component.
- **Adopt signals.** A future change can rewrite `BehaviorSubject`
  streams as `signal()` / `computed()`.
- **Zoneless change detection.** A future change can flip to
  `provideZonelessChangeDetection` once the streams are signals.
- **Add frontend unit tests.** A future change can add Vitest or
  keep Karma + add specs for `AppService` and `AuthService`.
- **Add a per-change CI pipeline.** Documented as a recommended
  contract in `006`.
