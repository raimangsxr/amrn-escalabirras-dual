# persistence-postgres Contract

## Purpose

Define the PostgreSQL persistence layer for the escalabirras backend.
This contract is the source of truth for the database schema, the
data-access conventions, and the migration workflow. It is owned by
change `002-backend-foundation` and consumed by the SQLAlchemy models
in `backend/app/models.py` and the Alembic migrations under
`backend/alembic/`.

## Engine and connection

- Database engine: PostgreSQL 16 (Docker image `postgres:16`).
- Driver: `psycopg[binary]` (psycopg 3).
- Connection string shape:
  `postgresql+psycopg://<user>:<password>@<host>:<port>/<database>`.
- Source of the connection string: the `DATABASE_URL` environment
  variable, loaded by `pydantic-settings` from `.env` (if present) or
  the process environment. Defaults in development to
  `postgresql+psycopg://escalabirras:escalabirras@localhost:5432/escalabirras`.
- For tests, the test suite uses a SQLite in-memory database
  (`sqlite:///:memory:`) to remain Docker-free. SQLite-specific
  differences (no `FOR UPDATE`, no `SERIAL`) are accepted as
  documented divergences; the production schema targets Postgres
  exclusively.

## Schema

Two tables in v1. The first holds tournament participants, the
second holds embed-token credentials. No seed data. The Top 3
leaderboard and the recent history are derived queries on
`participants`; no materialized view is required.

### `participants`

| Column | Type | Constraints |
| --- | --- | --- |
| `id` | `INTEGER` | `PRIMARY KEY`, generated by `SERIAL` / `IDENTITY` |
| `name` | `VARCHAR(20)` | `NOT NULL` |
| `crates` | `INTEGER` | `NOT NULL`, `DEFAULT 0`, `CHECK (crates >= 0)` |
| `created_at` | `TIMESTAMPTZ` | `NOT NULL`, `DEFAULT NOW()` |

Indexes:

- `ix_participants_created_at_desc` on `(created_at DESC)` to support
  `GET /v1/history`.
- `ix_participants_crates_desc` on `(crates DESC, created_at ASC)`
  to support `GET /v1/leaderboard/top`.

SQL DDL (Postgres):

```sql
CREATE TABLE participants (
    id          SERIAL PRIMARY KEY,
    name        VARCHAR(20) NOT NULL,
    crates      INTEGER NOT NULL DEFAULT 0 CHECK (crates >= 0),
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX ix_participants_created_at_desc
    ON participants (created_at DESC);

CREATE INDEX ix_participants_crates_desc
    ON participants (crates DESC, created_at ASC);
```

### `embed_tokens`

Introduced by change `004-auth-and-embed-tokens`. Holds the operator-
issued long-lived credentials used by the iframe to bypass login.

| Column | Type | Constraints |
| --- | --- | --- |
| `id` | `INTEGER` | `PRIMARY KEY`, generated by `SERIAL` |
| `name` | `VARCHAR(100)` | `NOT NULL`, human-readable label |
| `token_hash` | `VARCHAR(64)` | `NOT NULL`, `UNIQUE`, hex sha256 of the plaintext token |
| `created_at` | `TIMESTAMPTZ` | `NOT NULL`, `DEFAULT NOW()` |
| `last_used_at` | `TIMESTAMPTZ` | `NULL` until first use, updated on each successful redemption |
| `revoked_at` | `TIMESTAMPTZ` | `NULL` while active; soft-revoke sets it to `NOW()` |

Indexes:

- `ix_embed_tokens_token_hash` on `(token_hash)` to support the
  `WHERE token_hash = :h AND revoked_at IS NULL` lookup in
  `POST /v1/auth/embed-token`.

SQL DDL (Postgres):

```sql
CREATE TABLE embed_tokens (
    id            SERIAL PRIMARY KEY,
    name          VARCHAR(100) NOT NULL,
    token_hash    VARCHAR(64) NOT NULL UNIQUE,
    created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    last_used_at  TIMESTAMPTZ,
    revoked_at    TIMESTAMPTZ
);

CREATE INDEX ix_embed_tokens_token_hash ON embed_tokens (token_hash);
```

The plaintext token is **never** stored: it is shown to the operator
exactly once at creation time, hashed (sha256) before insert, and the
hash is what every later redemption compares against. `DELETE
/v1/tokens/{id}` is implemented as `UPDATE embed_tokens SET
revoked_at = NOW() WHERE id = :id`; rows are kept for audit.

## SQLAlchemy models

Defined in `backend/app/models.py`:

```python
class Participant(Base):
    __tablename__ = "participants"

    id: Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str] = mapped_column(String(20), nullable=False)
    crates: Mapped[int] = mapped_column(
        Integer, nullable=False, default=0, server_default="0"
    )
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True),
        nullable=False,
        server_default=func.now(),
    )


class EmbedToken(Base):
    __tablename__ = "embed_tokens"

    id: Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str] = mapped_column(String(100), nullable=False)
    token_hash: Mapped[str] = mapped_column(String(64), nullable=False, unique=True)
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), nullable=False, server_default=func.now()
    )
    last_used_at: Mapped[datetime | None] = mapped_column(DateTime(timezone=True))
    revoked_at: Mapped[datetime | None] = mapped_column(DateTime(timezone=True))
```

The `CHECK (crates >= 0)` constraint is enforced at the database
level; SQLAlchemy does not duplicate it. The application layer also
guards against underflow in `services/records.py` and returns
`409 crates_underflow` before issuing the UPDATE.

## Migrations

- Tool: Alembic 1.13+.
- Config: `backend/alembic.ini` and `backend/alembic/env.py`.
- `env.py` reads `DATABASE_URL` from the application settings and
  loads `target_metadata` from `Base.metadata`.
- Initial migration: `backend/alembic/versions/0001_create_participants.py`.
  Generated with `alembic revision --autogenerate -m "create
  participants"` and reviewed by hand to ensure indexes and the
  CHECK constraint are present.
- Auth migration: `backend/alembic/versions/0002_add_embed_tokens.py`,
  introduced by change `004-auth-and-embed-tokens`. Hand-written;
  creates the `embed_tokens` table and its `token_hash` index.
- Apply: `alembic upgrade head` from `backend/`. The `docker-compose`
  service runs this step before starting the API.

## Data-access conventions

- Every route handler depends on `get_db()` (a FastAPI dependency in
  `backend/app/deps.py`) which yields a `Session` and closes it on
  request completion.
- Mutations to `crates` always go through `crud.adjust_crates(...)`,
  which:
  1. Locks the target row with `SELECT ... FOR UPDATE` (Postgres only;
     no-op on SQLite).
  2. Computes `previous_max = MAX(crates) FROM participants WHERE id != target_id`
     inside the same transaction.
  3. Applies `+1` or `-1` and `flush()`es.
  4. Returns the updated `Participant` plus `is_new_record =
     (new_crates > previous_max or previous_max is None and new_crates > 0)`.
- Reads use `select(Participant)` with explicit `order_by` clauses
  matching the API contract.

## What is NOT persisted in v1

- No `events` or `tournaments` table; participants are flat.
- No `teams` table; team membership (red/blue) is a client-side
  concept (`currentParticipants[2]` on the frontend).
- No `users` / `operators` table; the operator password lives in the
  `ADMIN_PASSWORD` environment variable and is verified in-process,
  not in the database.
- No sessions table: post-login JWTs are self-validating (HS256
  signature + `exp` claim); there is no DB lookup per request.
- No `winner_history` table; the historical `app-core` contract's
  separate `winnerParticipants` array is collapsed into a derived
  query on `participants` (`ORDER BY crates DESC`).
- No audit log of who pressed what. `created_at` records when a
  participant was added; crate adjustments are not individually
  timestamped.
