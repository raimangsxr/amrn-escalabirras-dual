# api-rest Contract

## Purpose

Define the HTTP/REST contract exposed by the FastAPI backend under
`backend/` for the escalabirras app. This contract is the source of
truth for the public API surface; the implementation in `backend/app/`
must conform to it. The frontend in `src/` is the primary consumer of
this contract.

The backend is introduced by change `002-backend-foundation`. Until
change `003-migrate-storage` lands, the Angular frontend still uses
`localStorage` and does not call this API. This contract must remain
stable from `002` onward.

## Versioning

All routes are prefixed with `/v1`:

```
/v1/auth/login
/v1/auth/embed-token
/v1/auth/logout
/v1/auth/me
/v1/tokens
/v1/tokens/{token_id}
/v1/participants
/v1/participants/{participant_id}
/v1/participants/{participant_id}/crates/increment
/v1/participants/{participant_id}/crates/decrement
/v1/leaderboard/top
/v1/history
/healthz
```

The OpenAPI schema is served at `/docs` (Swagger UI) and `/redoc`. The
exact schema is generated by FastAPI from the Pydantic models in
`backend/app/schemas.py`; this document is the authoritative prose
description.

## Authentication

Introduced by change `004-auth-and-embed-tokens`. See
`specs/contracts/auth/contract.md` for the full auth contract.

- Every `/v1/*` endpoint **except** `/v1/auth/login`,
  `/v1/auth/embed-token`, and `/healthz` requires
  `Authorization: Bearer <session_token>`. The token is a JWT signed
  HS256 with claims `sub`, `iat`, `exp` (24 h).
- Missing or invalid token → `401 Unauthorized` with
  `{code: "unauthorized"}`.
- The frontend obtains a JWT via `/v1/auth/login` (operator password)
  or `/v1/auth/embed-token` (iframe bypass).
- The JWT is sent on every request from the Angular frontend's HTTP
  interceptor.
- Embed tokens are long-lived credentials issued via `/v1/tokens` and
  redeemed at `/v1/auth/embed-token` for a fresh JWT. They are
  revocable.

## Resources

### `Participant`

```json
{
  "id": 1,
  "name": "Ana",
  "crates": 3,
  "created_at": "2026-06-27T12:34:56.789Z"
}
```

| Field | Type | Description |
| --- | --- | --- |
| `id` | `integer` | Server-assigned monotonic id. |
| `name` | `string` | Trimmed, 1 to 20 characters. |
| `crates` | `integer` | Non-negative crate count. |
| `created_at` | `string` (ISO 8601, UTC) | Server-assigned creation timestamp. |

### Error body

Every non-2xx response returns the same JSON shape:

```json
{
  "detail": "Human-readable message",
  "code": "machine_readable_code"
}
```

| HTTP | `code` | When |
| --- | --- | --- |
| `401` | `unauthorized` | Missing or invalid `Authorization: Bearer` on a protected endpoint. |
| `401` | `invalid_password` | `POST /v1/auth/login` with wrong password. |
| `401` | `invalid_embed_token` | `POST /v1/auth/embed-token` with unknown or revoked token. |
| `404` | `participant_not_found` | `participant_id` does not exist |
| `404` | `token_not_found` | `token_id` does not exist (on `DELETE /v1/tokens/{id}`). |
| `409` | `crates_underflow` | Decrement called when `crates == 0` |
| `422` | `invalid_name` | Name empty, longer than 20, or not a string |
| `422` | `validation_error` | Any other FastAPI request validation error |
| `500` | `internal_error` | Unhandled server error |

## Endpoints

### Auth endpoints

#### `POST /v1/auth/login`

Exchange the operator password for a session JWT.

- Auth: **public**.
- Request body: `{ "password": string }`.
- Responses:
  - `200 OK` + `{ "session_token": string, "expires_at": ISO8601 }`.
  - `401 Unauthorized` + `invalid_password`.

The backend reads `ADMIN_PASSWORD` from its environment. There is
exactly one operator; usernames are not part of v1.

#### `POST /v1/auth/embed-token`

Redeem an embed token (created via `/v1/tokens`) for a session JWT.
Used by the iframe bootstrap.

- Auth: **public** (the embed token itself is the credential).
- Request body: `{ "token": string }` where `token` starts with
  `embed_`.
- Side effects: updates `embed_tokens.last_used_at` for the matching
  row (only if not revoked).
- Responses:
  - `200 OK` + `{ "session_token": string, "expires_at": ISO8601 }`.
  - `401 Unauthorized` + `invalid_embed_token` for unknown or
    revoked tokens.

#### `POST /v1/auth/logout`

Server-side no-op (JWTs are stateless). Returns `204`. The client
removes the JWT from `sessionStorage`.

- Auth: Bearer.

#### `GET /v1/auth/me`

Returns `{ "authenticated": true, "operator": "operator" }` when
called with a valid Bearer token. Useful for the frontend to confirm
that a stored JWT is still valid.

- Auth: Bearer.
- Responses:
  - `200 OK` + `{ "authenticated": true, "operator": "operator" }`.
  - `401 Unauthorized` + `unauthorized` when the JWT is missing or
    invalid.

### Token endpoints

#### `POST /v1/tokens`

Create a new embed token. The plaintext token is returned **once**
in the response; only its hash is stored. The frontend must show a
copy-to-clipboard UI and warn the operator that the token will not
be displayed again.

- Auth: Bearer.
- Request body: `{ "name": string, min_length=1, max_length=100 }`.
- Responses:
  - `201 Created` +
    `{ "id": int, "name": string, "token": "embed_...", "created_at": ISO8601 }`.

#### `GET /v1/tokens`

List all embed tokens (excluding their values).

- Auth: Bearer.
- Responses: `200 OK` +
  `{ "tokens": [{ "id": int, "name": string, "created_at": ISO8601, "last_used_at": ISO8601 | null, "revoked_at": ISO8601 | null }, ...] }`.

#### `DELETE /v1/tokens/{token_id}`

Soft-revoke an embed token. Sets `revoked_at = NOW()`.

- Auth: Bearer.
- Responses:
  - `204 No Content`.
  - `404 Not Found` + `token_not_found` when the id does not exist.

### Existing endpoints

#### `POST /v1/participants`

Create a new participant. Used by the frontend when the operator types
a name and presses Enter in a team input.

- Auth: Bearer.
- Request body: `{ "name": "Ana" }`
- Validation: `name` is trimmed; must be `1 <= len(name) <= 20`.
- Responses:
  - `201 Created` + `Participant` (with `crates: 0`).
  - `422 Unprocessable Entity` + `invalid_name` if validation fails.
- Side effects: inserts a row in `participants`; no other state
  changes.

#### `GET /v1/participants/{participant_id}`

Fetch a single participant by id.

- Auth: Bearer.
- Path parameter: `participant_id` (integer).
- Responses:
  - `200 OK` + `Participant`.
  - `404 Not Found` + `participant_not_found`.

#### `POST /v1/participants/{participant_id}/crates/increment`

Atomically add one crate to the participant. Returns whether the run
just set a new all-time record (i.e., the participant's `crates`
after the increment is strictly greater than every other
participant's `crates`). The comparison uses `SELECT ... FOR UPDATE`
inside the same transaction to avoid races between concurrent
clients.

- Auth: Bearer.
- Path parameter: `participant_id` (integer).
- Request body: none.
- Responses:
  - `200 OK` + `{ "participant": Participant, "is_new_record": boolean }`.
  - `404 Not Found` + `participant_not_found`.
- Edge case (first run ever): when the table is empty apart from the
  target row, `is_new_record` is `true` for the first increment. This
  is intentional: the first run is the inaugural record.

#### `POST /v1/participants/{participant_id}/crates/decrement`

Atomically subtract one crate from the participant. Clamped at zero:
a decrement on a participant with `crates == 0` is rejected with
`409 crates_underflow` instead of going negative.

- Auth: Bearer.
- Path parameter: `participant_id` (integer).
- Request body: none.
- Responses:
  - `200 OK` + `{ "participant": Participant, "is_new_record": false }`.
  - `404 Not Found` + `participant_not_found`.
  - `409 Conflict` + `crates_underflow`.

#### `GET /v1/leaderboard/top?limit=3`

Return the top N participants by crates descending. Ties are broken
by `created_at` ascending so the result is deterministic.

- Auth: Bearer.
- Query parameter: `limit` (integer, default `3`, min `1`, max `100`).
- Responses: `200 OK` + `Participant[]` (always returns at most `limit`
  entries; can be empty).

#### `GET /v1/history?limit=10&offset=0`

Return the most recent participants in reverse chronological order
(used by the `ParticipantListComponent`).

- Auth: Bearer.
- Query parameters:
  - `limit` (integer, default `10`, min `1`, max `100`).
  - `offset` (integer, default `0`, min `0`).
- Responses: `200 OK` + `Participant[]`. May be empty if there are
  no participants yet.

#### `GET /healthz`

Liveness probe. Used by `docker-compose` and the frontend's
optional health check.

- Responses: `200 OK` + `{ "status": "ok" }`.

## CORS

CORS is configured via the `CORS_ORIGINS` environment variable, a
comma-separated list of allowed origins. Default:
`http://localhost:4200`. The backend allows only `GET`, `POST`,
`OPTIONS`, and `DELETE` methods and only the `Content-Type` and
`Authorization` headers. Credentials are not supported in v1.

In production, `CORS_ORIGINS` must be the public origin(s) of the
escalabirras frontend (e.g. `https://app.example.com`). The parent
iframe does **not** need to be in `CORS_ORIGINS` because the iframe
loads the escalabirras frontend from the same origin as the API; the
iframe's JavaScript makes cross-origin requests only to its own
backend (same origin).

## Production security headers

Added by change `006-production-readiness`. Every response carries:

- `Content-Security-Policy: frame-ancestors <FRAME_ANCESTORS>`
  (default `*`, set to the parent app's origin in production).
- `X-Frame-Options: ALLOW-FROM <FRAME_ANCESTORS>` (legacy fallback).
- `X-Content-Type-Options: nosniff`.
- `Referrer-Policy: no-referrer`.
- `Strict-Transport-Security: max-age=63072000; includeSubDomains`
  (production only; suppressed when the request is over HTTP to
  avoid misleading headers behind an internal load balancer).

These are added by `app/middleware/security_headers.py`.

## Concurrency

- All mutations to `participants.crates` use `SELECT ... FOR UPDATE`
  on the target row, plus a read of `MAX(crates)` of the other rows
  inside the same transaction, to compute `is_new_record` atomically.
- Two clients incrementing the same participant at the same time will
  be serialized by the row lock. The first to acquire the lock
  determines the post-increment crates for both, and only one of
  them observes `is_new_record: true` if the post-increment value
  beats the prior max.
- Two clients incrementing different participants at the same time
  do not block each other (different rows).

## Pagination and ordering

- `GET /v1/leaderboard/top` is ordered by `crates DESC, created_at ASC`.
- `GET /v1/history` is ordered by `created_at DESC, id DESC`.
- Cursor-based pagination is intentionally not exposed in v1; the
  append-only dataset grows linearly with the number of runs in a
  single event and stays small in practice.

## What is NOT in this contract (yet)

- No authentication (v1 is single-operator, private network).
- No multi-event/tournament scoping (`participant` rows are flat).
- No `DELETE` or `PUT` (no edits after creation, only crate
  increments/decrements).
- No server-push (no SSE / WebSocket). The frontend polls if it
  needs live updates.
- No soft delete or archival.
- No rate limiting.
