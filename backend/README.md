# Escalabirras Backend

FastAPI + SQLAlchemy 2.0 + Alembic + PostgreSQL backend for the
escalabirras tournament app. Owned by SDD change
`002-backend-foundation`.

## Stack

- Python 3.11+
- FastAPI 0.115
- SQLAlchemy 2.0 (typed `Mapped[...]` models)
- Alembic 1.13 for migrations
- psycopg 3 (`psycopg[binary]`)
- pydantic v2 + pydantic-settings
- pytest + httpx for tests

## Quickstart (Docker)

```bash
docker compose up db -d
cp .env.example .env       # adjust DATABASE_URL if needed
python3 -m venv .venv
source .venv/bin/activate
pip install -e backend[dev]
cd backend
alembic upgrade head
uvicorn app.main:app --reload
```

The API is then served on `http://localhost:8000`, the Swagger UI on
`http://localhost:8000/docs`. Postgres is reachable on
`localhost:5432` (or `db:5432` from inside other containers). Adminer
(when enabled with `docker compose up adminer`) is on
`http://localhost:8080`.

## Quickstart (bare-metal Postgres)

## Tests

```bash
pytest backend -v
```

Tests run against an in-memory SQLite database (one per test, dropped
on teardown). They do **not** require Docker or Postgres. The
SQLite-vs-Postgres divergences are documented in
`specs/contracts/persistence-postgres/contract.md`:

- `with_for_update()` is a no-op on SQLite; the row lock is enforced
  on Postgres only.
- SQLite ignores the `TIMESTAMPTZ` timezone and stores naive datetimes.
  Pydantic still serializes to ISO 8601.

If you want full Postgres coverage, point `DATABASE_URL` at a test
database and run:

```bash
DATABASE_URL=postgresql+psycopg://... pytest backend -v
```

## Environment variables

| Name | Required | Default | Description |
| --- | --- | --- | --- |
| `DATABASE_URL` | no | `postgresql+psycopg://escalabirras:escalabirras@localhost:5432/escalabirras` | SQLAlchemy URL. |
| `CORS_ORIGINS` | no | `http://localhost:4200` | Comma-separated allowed origins. |
| `ADMIN_PASSWORD` | yes | — | Operator password used by `POST /v1/auth/login`. Compared in-process with constant-time equality. |
| `JWT_SECRET` | yes | — | HS256 signing secret. **Must be at least 32 characters** of high-entropy data. The backend refuses to start otherwise. Rotate to invalidate all existing sessions. |
| `JWT_TTL_SECONDS` | no | `86400` | Session JWT lifetime (default 24 h). |
| `FRAME_ANCESTORS` | no | `*` | CSP `frame-ancestors` and `X-Frame-Options: ALLOW-FROM` value. Set to the parent app's origin in production (e.g. `https://parent.example.com`). |

A `.env` file in `backend/` (or the repo root) is loaded
automatically by pydantic-settings.

### Generating a JWT secret

```bash
python3 -c "import secrets; print(secrets.token_urlsafe(48))"
```

## Migrations

```bash
alembic revision --autogenerate -m "..."  # creates a new version
alembic upgrade head                      # apply all migrations
alembic downgrade -1                      # roll back one step
```

`alembic/env.py` reads `DATABASE_URL` from the application settings,
so the same `.env` drives both the API and the migrations. The initial
migration is `0001_create_participants`; it creates the table with a
`CHECK (crates >= 0)` constraint and the two indexes described in
`specs/contracts/persistence-postgres/contract.md`.

## What is not here yet

- Multi-event / tournament scoping.
- Server-push (SSE / WebSocket).
- Rate limiting on `/v1/auth/login` (delegate to a WAF in front of
  the API in production).
- A production Dockerfile for the API (only the dev compose is
  shipped in `002-backend-foundation`).
- Testcontainers-based integration tests against real Postgres.
